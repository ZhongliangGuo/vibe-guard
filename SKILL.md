# Vibe Guard — AI-Aware Code Security Auditor

Vibe coding (using AI agents to generate code via natural language) introduces a unique threat surface: the code *looks* correct and passes cursory review, but may contain subtle backdoors, data exfiltration logic, or insecure patterns that a human would miss during rapid iteration. Vibe Guard detects these threats.

## Auto-Scan Mode

Vibe Guard supports two modes:

- **Manual mode** (default): The user explicitly asks for a security check.
- **Auto-scan mode**: Activated when the user says "enable auto security check" or "enable vibe-guard auto mode". Once enabled, perform a lightweight scan after every code generation or modification task. The user can disable it by saying "disable auto security check" or "disable vibe-guard auto mode".

In auto-scan mode, keep the check fast and focused — flag only Medium severity and above. In manual mode, perform a full deep scan including Low severity findings.

## Detection Pipeline

When scanning code, follow this pipeline in order:

### Step 1: Identify the Environment

Determine the programming language(s), version(s) if detectable, and dependency files present:

- Look for version specifiers: `python_requires`, `Pipfile`, `.python-version`, `engines` field in `package.json`, `go.mod`, `Cargo.toml`, `pom.xml`, `.java-version`, etc.
- Look for dependency manifests: `requirements.txt`, `Pipfile.lock`, `package.json`, `package-lock.json`, `yarn.lock`, `pom.xml`, `build.gradle`, `go.sum`, `Cargo.lock`, `composer.json`, etc.

### Step 2: Load Relevant References

Based on the environment detected in Step 1, read the appropriate reference files from the `references/` directory within this skill's folder:

- **Language-specific**: Read `references/languages/<language>/<version-range>.md` matching the detected language and version. If the exact version range file doesn't exist, read the closest match or the general `common.md` for that language.
- **Dependency-specific**: Read `references/dependencies/<ecosystem>/<category>.yaml` for the relevant package ecosystem.

These references contain known vulnerabilities, dangerous APIs, and version-specific gotchas that serve as ground truth for detection. The references are designed to be updated independently — always trust them as the authoritative source.

### Step 3: Universal Pattern Scan

Scan the code against ALL of the following universal threat categories. These apply regardless of language:

#### 3.1 Data Exfiltration
- Outbound HTTP/HTTPS requests to hardcoded external URLs or IPs
- DNS-based data tunneling (encoding data in DNS queries)
- Covert channels via seemingly innocent APIs (e.g., error reporting endpoints that leak data)
- Requests carrying sensitive data (env vars, tokens, file contents) in headers, query params, or body
- WebSocket connections to unknown endpoints
- Exfiltration via email/SMTP calls embedded in code

#### 3.2 Remote Code Execution (RCE)
- Dynamic code evaluation: `eval()`, `exec()`, `Function()`, `setTimeout(string)`, `setInterval(string)`
- Subprocess/shell calls with user-controlled or dynamically constructed arguments
- Deserialization of untrusted data (pickle, yaml.load, ObjectInputStream, unserialize)
- Template injection (server-side template engines with user input)
- Code loading from remote URLs at runtime

#### 3.3 Credential & Secret Exposure
- Hardcoded API keys, tokens, passwords, connection strings
- Secrets in code comments, variable names like `password`, `secret`, `api_key`, `token`
- Private keys or certificates embedded in source
- `.env` file contents hardcoded instead of referenced
- Credentials logged to stdout/stderr or written to files

#### 3.4 Supply Chain & Dependency Risks
- Typosquatting packages (names similar to popular packages but slightly different)
- Pinned versions known to be vulnerable (cross-reference with loaded dependency references)
- Lock file manipulation (mismatched lock file vs manifest)
- Post-install scripts in dependencies that execute arbitrary code
- Dependencies pulled from non-standard registries
- Wildcard or unpinned dependency versions allowing silent upgrades

#### 3.5 Backdoor Authentication & Access Control
- Hardcoded admin credentials or "god mode" accounts
- Authentication bypass logic (always-true conditions, commented-out checks)
- Hidden API endpoints not documented or exposed in routing
- JWT/session token manipulation (weak signing, no expiry, algorithm confusion)
- Privilege escalation patterns (unchecked role assignments)

#### 3.6 Obfuscated & Suspicious Code
- Base64-encoded strings that decode to URLs, IPs, or executable code
- Character code construction (`String.fromCharCode`, `chr()`, hex escapes)
- Variable names designed to mislead (`safeCheck` that actually disables checks)
- Unusually complex one-liners that obscure intent
- Code that looks auto-generated or copy-pasted from unfamiliar sources
- Steganographic payloads hidden in data structures

#### 3.7 File System Abuse
- Reading sensitive system files (`/etc/passwd`, `~/.ssh/`, `~/.aws/credentials`)
- Writing to system directories or startup scripts
- Path traversal patterns (`../`, symlink following)
- Temporary files with sensitive data and insecure permissions
- Unauthorized modification of configuration files

#### 3.8 Network & Infrastructure Threats
- Reverse shell patterns (socket connections with stdin/stdout redirection)
- SSRF vectors (server-side requests to internal IPs, cloud metadata endpoints like `169.254.169.254`)
- Port scanning or network enumeration code
- Proxy/tunnel setup (SOCKS, HTTP tunneling)
- Overly permissive CORS configurations (`Access-Control-Allow-Origin: *`)
- Exposed debug endpoints or admin panels

#### 3.9 Persistence & Stealth
- Cron jobs, scheduled tasks, or startup hooks being installed
- Self-replicating code or auto-update mechanisms from external sources
- Log suppression or tampering (deleting logs, filtering security events)
- Anti-debugging or anti-analysis techniques
- Delayed execution (time bombs, conditional triggers)
- Process hiding or name spoofing

#### 3.10 Cryptographic Weaknesses
- Weak algorithms (MD5/SHA1 for security purposes, DES, RC4)
- Hardcoded IVs, salts, or keys
- Predictable random number generation for security contexts (`Math.random()`, `random` without `secrets`)
- Missing or improper certificate validation (SSL verify disabled)
- Custom crypto implementations (rolling your own crypto is almost always wrong)

#### 3.11 Input Validation & Injection
- SQL injection vectors (string concatenation in queries)
- Command injection (unsanitized input in shell commands)
- XSS vectors (unsanitized user input in HTML output)
- LDAP, XML, or XPath injection
- Header injection (CRLF injection in HTTP headers)
- Missing input bounds checking

#### 3.12 AI/Vibe-Coding-Specific Risks
These are patterns particularly common in AI-generated code that a human developer might not produce:

- Prompt injection via code comments (comments designed to manipulate AI agents reading the code)
- Overly permissive configurations that "just work" but are insecure (e.g., `chmod 777`, `--no-verify`, disabling CSRF)
- Plausible-looking but subtly wrong security logic (e.g., checking authentication but not authorization)
- Missing error handling that silently swallows security-critical failures
- Debug/development configurations left enabled (Flask debug mode, Django DEBUG=True, verbose logging of secrets)
- Copy-paste vulnerabilities from training data (known-vulnerable code patterns that appear frequently in AI training sets)
- Excessive permissions requests (requesting admin/root when not needed)
- Mock/stub security functions that appear functional but do nothing

### Step 4: Cross-Reference with References

After the universal scan, cross-reference findings with the language and dependency reference files loaded in Step 2. This catches version-specific issues that the universal patterns wouldn't detect — for example, a Python 2 `input()` call that executes arbitrary code, or a specific version of `lodash` with a prototype pollution vulnerability.

### Step 5: Severity Classification

Classify each finding:

| Level | Icon | Criteria | Example |
|-------|------|----------|---------|
| Critical | 🔴 | Active exploitation risk, data breach, or RCE with minimal effort | Reverse shell, hardcoded admin backdoor |
| High | 🟠 | Exploitable vulnerability requiring some conditions | SQL injection, disabled SSL verification |
| Medium | 🟡 | Security weakness that could be exploited in combination with other issues | Weak crypto, overly permissive CORS |
| Low | 🔵 | Code smell or best-practice violation with limited direct risk | Missing input length limits, verbose error messages |

## Output Format

ALWAYS produce output in **both** formats:

### Format 1: Inline Annotated Code

Show the relevant code sections with security annotations inserted as comments directly above or beside the problematic lines. Use the severity icons as markers:

```
# 🔴 CRITICAL: Hardcoded admin backdoor — remove this entirely
if username == "superadmin" and password == "master123":
    return grant_full_access()

# 🟠 HIGH: SQL injection — use parameterized queries instead
query = f"SELECT * FROM users WHERE id = {user_input}"

# 🟡 MEDIUM: Debug mode enabled — disable in production
app.run(debug=True, host='0.0.0.0')
```

## Important Reminders

- **False positive awareness**: Not every `eval()` is malicious, not every outbound request is exfiltration. Consider the context — a testing framework using `eval` is different from user input flowing into `eval`. When uncertain, flag it as a finding but note the ambiguity.
- **Prioritize actionability**: Every finding should include a concrete remediation. "Don't do this" is not helpful — show what to do instead.
- **Vibe coding context**: Remember that in vibe coding, the developer may not fully understand the generated code. Explanations should be clear enough for someone who didn't write the code themselves.
- **Stay current**: The reference files in `references/` are the ground truth for known vulnerabilities. Always load and check them. If you detect a language or dependency not covered by existing references, still apply universal patterns and note that specific reference coverage is unavailable.
